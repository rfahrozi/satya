Blueprint Arsitektur Final
Chatbot PTSP WhatsApp

Kerangka pembangunan Asisten Virtual PTSP berbasis WhatsApp, memadukan kebutuhan operasional peradilan dengan arsitektur microservices RAG (Retrieval-Augmented Generation) lokal.

1

Tujuan Sistem

Sistem ini ditujukan untuk menjadi asisten virtual PTSP yang melayani masyarakat melalui WhatsApp, dengan jawaban yang bersumber dari dokumen resmi pengadilan melalui pendekatan RAG. Prinsip utamanya adalah:

  • Mudah diakses publik melalui platform WhatsApp.
  • Jawaban harus mengacu ke SOP/aturan internal, bukan sekadar pengetahuan umum model AI.
  • Terdapat mekanisme eskalasi (handoff) ke petugas manusia.
  • Data sensitif harus dibatasi/disensor sebelum diproses oleh AI.
  • Sistem bisa dimulai dari Proof of Concept (PoC) lokal, lalu dinaikkan skalanya ke tingkat produksi.
2

Prinsip Arsitektur Inti & Topologi

Fondasi teknis menggunakan arsitektur microservices berbasis Docker dalam satu jaringan virtual lokal dengan komponen inti: PostgreSQL + pgvector (Vector Store), MinIO (Object Storage), Redis (Cache/Queue), FastAPI (Backend API), Ollama (AI Engine), dan OpenClaw (Orchestrator).

3

Lapisan Arsitektur (Layering)

3.1. Channel Layer

Lapisan antarmuka pengguna, menggunakan WhatsApp Business API (WABA) melalui BSP resmi Meta.

  • Menerima & mengirim pesan.
  • Meneruskan event ke webhook backend.

Pending Design Vendor BSP, format event, dan mekanisme approval Meta masih perlu dipilih.

3.2. Application/API Layer

Backend API (Python/FastAPI). Bertindak sebagai otak operasional. Dibagi menjadi modul logis:

  • Webhook Handler
  • Conversation Service
  • RAG Service
  • Document Ingestion Service
  • Security & Sanitization
  • Escalation Service

3.3. Workflow Layer

OpenClaw bertindak sebagai control plane alur bisnis (mesin keputusan proses).

  • Menentukan apakah cukup dijawab dari FAQ statis.
  • Menentukan apakah butuh jalur RAG (SOP).
  • Menentukan apakah harus dialihkan (handoff) ke petugas.
4

Lapisan Data

🗄️ PostgreSQL + pgvector

Pusat data terpadu untuk domain operasional dan knowledge retrieval.

  • Log percakapan & sesi pengguna.
  • Vector embeddings dokumen SOP.

Pending Design Struktur tabel, indeks, relasi, dan strategi partisi belum dirinci.

📦 MinIO Storage

Single Source of Truth untuk file mentah.

  • Menyimpan dokumen asli (SOP PTSP, Surat Edaran, Aturan Biaya).

⚡ Redis

Cache dan Message Broker / Queue.

  • Antrean proses ingestion dokumen.
  • Buffering chunking/embedding agar server tidak crash.
  • Penyimpanan state sementara workflow.
5

Lapisan AI dan Pipeline RAG

Ollama sebagai AI Engine Lokal

Menjalankan dua fungsi terpisah secara sehat: Embedding Model untuk mengubah teks menjadi vektor (indexing/retrieval), dan Generative Model (LLM seperti Llama-3 8B) untuk menyusun jawaban. RTX 4070 (8GB VRAM) valid untuk PoC ini.

Pipeline RAG Final

A. Ingestion Dokumen

Admin unggah PDF SOP. Disimpan di MinIO. Backend Python mengekstrak teks (PyPDF2/pdfplumber).

B. Chunking

Teks dibagi menjadi potongan 500–1000 token dengan overlap agar konteks hukum tidak terputus.

C. Embedding

Teks dikirim ke model embedding Ollama. Vektor disimpan ke PostgreSQL + pgvector.

D. Retrieval

Pertanyaan user diubah jadi vektor. DB melakukan cosine similarity search mencari ~3 chunk paling relevan.

E. Generation

Chunk relevan digabung dengan pertanyaan ke dalam Prompt, dikirim ke LLM generatif Ollama untuk dijawab.

6

Human Handoff & Batas Kemampuan Bot

Human handoff adalah jalur inti, bukan sekadar fitur tambahan. Trigger handoff wajib:

  • Pertanyaan terlalu spesifik terkait perkara yang sedang berjalan.
  • Bot tidak tahu jawaban (confidence rendah / dokumen tidak ada).
  • Pertanyaan membutuhkan tindakan administratif langsung dari petugas PTSP asli.

Pending Design Cara admin menerima limpahan chat (Dashboard, CRM, atau nomor WA khusus) dan SLA respons.

7. Keamanan & Privasi

Wajib Ada: Disclaimer komunikasi dengan AI & Masking sederhana (NIK, No. Rekening) sebelum data diproses.

Pending Design Enkripsi at-rest, secret management, retensi log, audit trail admin, klasifikasi data.

8. Observability & Monitoring

Wajib Dipantau: Status container, antrean Redis, beban GPU Ollama, rasio sukses retrieval, jumlah kegagalan WABA, rasio handoff admin.

Pending Design Stack tools spesifik (misal: Prometheus, Grafana, ELK).

9. Blueprint Deployment

Mode Lokal / PoC: Seluruh komponen berjalan via docker-compose. Sangat hemat biaya, aman, dan cocok untuk spesifikasi laptop RTX 4070 / 32GB RAM.

Mode Produksi: Desain microservices ini secara inheren siap dimigrasi ke cloud. Pending Design Topologi detail cloud/server production.

10

Urutan Interaksi Utama Sistem

Alur A — Pertanyaan Masyarakat

  1. Kirim pesan ke WA PTSP.
  2. WABA teruskan event ke Webhook Backend.
  3. Backend verifikasi, kelola sesi, dan masking data sensitif.
  4. OpenClaw tentukan rute.
  5. Backend embed pertanyaan via Ollama.
  6. PG+pgvector cari chunk relevan.
  7. Backend susun Prompt.
  8. Ollama hasilkan jawaban.
  9. Balasan dikirim via WABA.
  10. Jika ragu -> Handoff ke Admin.

Alur B — Ingest Dokumen Resmi

  1. Admin unggah PDF SOP.
  2. File masuk ke MinIO.
  3. Backend tarik dan ekstrak teks.
  4. Teks dipecah (chunking) + overlap.
  5. Ollama lakukan embedding tiap chunk.
  6. Vektor disimpan ke PG+pgvector.
  7. Dokumen siap digunakan AI.

Roadmap Pembangunan Aman

Fase 1: Fondasi Lokal

Bangun stack Docker lokal (PG, MinIO, Redis, FastAPI, Ollama, OpenClaw). Fokus pada ingest dokumen, retrieval, dan generation.

Fase 2: Knowledge Base

Kumpulkan SOP, FAQ, biaya panjar. Uji kualitas chunking, embedding, dan akurasi retrieval dokumen hukum.

Fase 3: Chatbot Core

Tambahkan session management, disclaimer otomatis, masking data sensitif NIK/Rekening, dan rules bot.

Fase 4: Human Handoff

Bangun alur eskalasi ke admin PTSP untuk pertanyaan spesifik perkara berjalan atau yang tidak terjawab AI.

Fase 5: Integrasi WA

Hubungkan backend FastAPI ke WABA/BSP. Uji coba alur percakapan End-to-End dengan pengguna nyata.

Fase 6: Hardening

Lengkapi logging, monitoring (Observability), kontrol akses admin, retensi data, persiapan migrasi ke cloud.

Kesimpulan Blueprint

"Chatbot PTSP WhatsApp dibangun sebagai microservices RAG system berbasis Docker, dengan WhatsApp Business API sebagai channel publik, FastAPI sebagai backend utama, OpenClaw sebagai workflow orchestrator, PostgreSQL + pgvector sebagai pusat data relasional dan vector retrieval, MinIO sebagai penyimpanan dokumen sumber, Redis sebagai cache/queue, dan Ollama sebagai engine AI lokal. Alur dimulai dari pesan WhatsApp, sanitasi, routing, retrieval SOP, generasi jawaban, hingga eskalasi admin jika dibutuhkan."
