Tentang Sistem SATYA — Monev

Sistem Monitoring dan Evaluasi Terpadu ini dirancang sebagai instrumen strategis penyelenggaraan tugas administrasi peradilan di lingkungan Pengadilan Tinggi Kepulauan Riau. Sistem ini memiliki peran ganda: (1) Monitoring Eksternal untuk pelaporan 12 jenis laporan wajib dari seluruh Satker (PN) ke PT, dan (2) Monitoring Internal untuk evaluasi mandiri pemenuhan dokumen dari tiap bagian di dalam PT sendiri guna mendukung standar AMPUH, PMPZI, dan AKIP.

✓ Sertifikasi AMPUH ✓ Zona Integritas (PMPZI) ✓ SAKIP / AKIP ✓ Regulasi MA & PT

Peran & Tanggung Jawab Aktor

Tiga peran utama dalam ekosistem monitoring

👨‍⚖️
Pimpinan PT
Pengawas Utama
  • ›Memantau dashboard kepatuhan agregat melalui Heatmap 12 Bulan
  • ›Mengunduh rekapitulasi nilai dan kepatuhan dalam format PDF & Excel
  • ›Melihat laporan yang sudah diverifikasi dan dinilai mutunya oleh Admin
  • ›Menggunakan hasil statistik sebagai dasar pembinaan (reward & punishment)
🛡️
Admin PT
Pengelola Sistem
  • ›Membuat akun pengguna Satker & mengatur master data periode aktif
  • ›Memeriksa kelengkapan berkas dan memberikan status laporan
  • ›Memberikan Penilaian Mutu (Scoring 0-100) sebagai standar evaluasi AMPUH
  • ›Melacak dan mencatat riwayat perbaikan pada sistem Audit Trail
🏢
Satker (PN)
Pelapor Satuan Kerja
  • ›Mengunggah laporan wajib dalam format PDF dan data pendukung (Excel/Word) pada periode aktif
  • ›Menerima pemberitahuan revisi instan melalui In-App Notification
  • ›Melihat catatan verifikator dari sistem Audit Trail History
  • ›Memastikan kelengkapan & mutu berkas untuk mendapat nilai terbaik
🏢
Internal PT & Verifikator
Uploader, Approver, Verifier
  • ›Koordinator Proses (Uploader): Kasubbag, Panmud, Staff, dan Panitera Pengganti mengunggah bukti dokumen di Portal Checklist PT.
  • ›Pejabat Eselon III (Approver): Panitera, Sekretaris, dan Kabag memantau progres dan melakukan Approval dokumen unit di bawahnya.
  • ›Verifikator (Tim Mutu/Arsiparis): Melakukan verifikasi akhir setelah di-approve, memberi nilai/revisi.
  • ›Viewer: Berhak memonitor seluruh dokumen di Portal Checklist tanpa hak mengubah dokumen.
🛡️
Security & SRE (High Availability)
Reliabilitas v2.1.0
  • ›Circuit Breaker (Opossum): API tetap melayani antrean secara Fail-Fast saat MinIO down tanpa menghanguskan server.
  • ›Disk Storage & Caching: Menghindari OOM Crash dengan Redis Cache dan Streaming File dari HDD langsung ke MinIO.
  • ›Proteksi OWASP Terkini: Validasi Magic Bytes MimeType, Rate Limiter Lupa Password, Pencegahan Path Traversal, hingga JWT Blacklisting.
  • ›Sentry & Prometheus: Endpoint `/health` mendeteksi ketersediaan Postgres dan Sentry Auto-Instrumentation membaca metrik slow query.

1
Pembuatan Akun Satker
Admin PT membuat akun untuk setiap Satuan Kerja (Pengadilan Negeri) beserta informasi email notifikasi yang valid. Setiap akun diassign role dengan tingkatan masing-masing (contoh: ADMIN_PN, KPN, SATKER_PN, dsb).
POST /auth/register Role: ADMIN_PT / ADMIN_PN
2
Konfigurasi Periode Aktif
Admin menetapkan periode pelaporan aktif (Bulanan, Triwulan, Semester, atau Tahunan) beserta deadline. Sistem otomatis mengunci akses upload di luar periode aktif.
POST /periods 4 Tipe Periode
3
Upload Laporan oleh Satker
Satker mengunggah laporan wajib dalam format PDF dan data pendukung format (XLSX, XLS, DOC, DOCX) melalui portal khusus. Sistem memvalidasi MIME type, ukuran file (<10MB), dan menyimpan di MinIO Object Storage.
POST /reports/upload MinIO Storage
4
Verifikasi oleh Admin
Admin mengunduh dan memeriksa setiap laporan melalui pre-signed URL. Status dapat diubah menjadi Lengkap, Revisi, atau Ditolak disertai catatan.
PATCH /reports/:id/verify Pre-Signed URL 1 jam
5
Notifikasi & Revisi
Jika status adalah Revisi, BullMQ Worker mengirim email notifikasi otomatis ke Satker beserta catatan Admin. Satker mengunggah ulang dokumen yang diperbaiki.
BullMQ Worker Nodemailer + SMTP
6
Pemantauan Pimpinan
Pimpinan PT memantau dashboard agregat yang menampilkan persentase kepatuhan per Satker, jumlah laporan lengkap vs. terlambat, dan tren historis.
GET /dashboard-agregat Computed SQL ~20ms
✓
Evaluasi & Penutupan Periode
Admin menutup periode aktif. Sistem mengunci semua upload, menghasilkan rekapitulasi akhir, dan data menjadi bahan evaluasi kinerja Sertifikasi AMPUH.
PATCH /periods/:id/close Rekap PDF Otomatis

Alur Data Teknis — Upload Report Flow

① Autentikasi & Otorisasi
Satker login → POST /auth/login → Bcrypt verify → JWT issued (15m) + Refresh Token (7d) disimpan di httpOnly cookie. Middleware memvalidasi role sebelum setiap request.
② Upload Laporan
Satker upload file → POST /reports/upload → Multer validasi MIME + size → file disimpan ke MinIO bucket dengan path satker_id/periode/report_type_id/filename → metadata disimpan ke PostgreSQL.
③ Verifikasi Admin & Scoring
Admin minta URL → GET /reports/:id/download → Admin memberikan nilai 0-100 dan status → PATCH /reports/:id/verify → data tercatat di Audit Trail → In-App Notification memberikan alert real-time ke UI Satker.
④ Cron Job Reminder Email
BullMQ Worker scan DB setiap hari → identifikasi Satker yang belum upload pada H-3, H-1, Hari-H → batch email dikirim via Nodemailer SMTP → log pengiriman disimpan di tabel notification_logs.

Sistem Notifikasi & Reminder

In-App Alerts, Audit Trail, dan Notifikasi Email Bertahap

H-3
Pengingat Awal — 3 Hari Sebelum Deadline
Email dikirim ke semua Satker yang belum mengunggah seluruh laporan. Pesan berisi daftar laporan yang masih kosong dan link langsung ke portal upload. Tujuan: memberikan waktu cukup untuk mempersiapkan dokumen.
H-1
Pengingat Kritis — 1 Hari Sebelum Deadline
Email mendesak dikirim ulang ke Satker yang masih memiliki laporan dengan status Belum Upload atau Revisi. Subjek email menggunakan kata kunci urgensi dan menyertakan nama pimpinan sebagai CC.
H-0
Notifikasi Deadline — Hari-H Penutupan
Pada hari deadline, sistem mengirim notifikasi final. Setelah pukul 23:59, periode dikunci otomatis dan laporan yang belum masuk secara otomatis ditandai sebagai Terlambat di database.
REVISI
Notifikasi Revisi — Dikirim Segera
Ketika Admin menandai laporan sebagai Revisi, BullMQ Worker langsung mem-push job email ke antrian Redis. Email berisi catatan revisi spesifik dari Admin dan deadline perbaikan yang ditetapkan.

Status Rilis (v2.1.0) & Peningkatan Mendatang

Rekapitulasi perbaikan sistem terbaru & technical debt

✅ Selesai (v2.1.0)
Integrasi Modul Internal PT (AMPUH/PMPZI/AKIP)
Sebanyak 295 Checklist master data telah dimuat dan di-mapping kepada 15 Jabatan Resmi. Internal PT dapat mengunggah dokumen melalui Portal Checklist Internal dan termonitor di Dashboard Eksekutif. Dilengkapi juga dengan Management Review, Risk Heatmap interaktif, dan tracking Activity Logs (Audit Trail).
✅ Selesai (v2.1.0)Bug Fix
Perbaikan Bug Kritis Ekosistem
Sistem email notifikasi telah diperbaiki (sebelumnya salah tujuan ke username, sekarang ke alamat email). Storage dokumen telah tersinkronisasi 100% menggunakan MinIO S3-compatible, memblokir limitasi ukuran. Skema database telah dimigrasi (`period_type` dan `period_unit`) untuk mendukung Periode Semester dan Tahunan secara dinamis. Pagination dan Rate Limiting (10x-15x per menit) juga telah diaktifkan untuk API upload.
🟡 Rencana Lanjutan
Konfigurasi Deadline Per Jenis Laporan (Custom Deadline)
Saat ini deadline global menggunakan DEADLINE_DAY=10 (tanggal 10 setiap bulannya). Di rilis mendatang akan diimplementasikan UI untuk tabel deadline_configs agar setiap jenis laporan dapat di-override dengan tenggat berbeda sesuai standar masing-masing regulasi (misal: laporan Komdanas paling lambat tanggal 5).
